# Meowdoku 性能与结构调优清单

> 审查范围：`Assets/Scripts/` 全量脚本 + `BlockPrefab.prefab` / `TopUI.prefab` / `SampleScene.unity` 组件构成。
> 本文档仅输出可调优点，**不修改任何代码**。每条标注：位置 / 现象 / 影响 / 建议。
> 优先级：🔴 高（真实热点） / 🟠 中（值得做） / 🟡 低（锦上添花）/ 🔵 结构性（非性能）。

---

## 0. 现状评估（已做得好的，避免误改）

| 项 | 现状 | 评价 |
|---|---|---|
| Canvas 动静分离 | `Canvas_Static_Grid`(格子,SO=1) / `Canvas_Dynamic_HUD`(HUD+按钮,SO=2) 已拆 | ✓ 正确，格子顶点脏不会波及 HUD 文本 |
| 对象池 | `blockPool` 复用格子，切关不 Instantiate/Destroy | ✓ |
| 显隐方式 | `Image.enabled` 切显隐而非 `SetActive`（不破坏顶点拓扑） | ✓ 注释已说明，是高频滑动低耗的关键 |
| cat/cross RaycastTarget | 已关 | ✓ 装饰节点不参与 raycast |
| 计数 O(1) | `currentCatCount` 替代实时 `FindAll` | ✓ |
| 延迟判定模型 | pendingToggle 协程，双击取消无闪烁 | ✓ 逻辑成熟 |
| 解算单一真相源 | `MapSolver` 供 Editor 烘焙与运行时回退共用 | ✓ |
| 反馈分层 | `FeedbackManager` / `NativeHapticDriver` / `TweenRunner` 三层 | ✓ |

> **结论**：高频滑动路径（PointerEnter → RefreshVisual → Image.enabled）本身很轻，普通 Image 的 `OnPopulateMesh` 无 GC。真正值得调优的点集中在**原生触觉 JNI 开销**、**raycast 遍历**、**切关时 RoundedImage 的 GC 峰**、**通关后的输入锁**几处。

## 一、运行时性能（高频路径）

### 🔴 1.1 Android 触觉每次震动重复 new AndroidJavaClass / AndroidJavaObject  ✅ 已实施
- **位置**：`NativeHapticDriver.AndroidTrigger` / `EnsureReady`
- **改动**：把 SDK_INT、`currentActivity`、`Vibrator` 实例、`VibrationEffect` 类与 4 个预设 effect（不可变）缓存为静态字段；首次触发在 `EnsureReady()` 一次性初始化，之后每次 `Trigger` 仅一次 `vibrate(effect)` JNI 调用、零分配。VibrationEffect 创建失败保持 null 时回退 `vibrate(long)`。增加 `_disabled` 标志：触发/初始化异常后置位，避免高频滑动刷屏 LogError（app 重启复位）。
- **结果**：滑动每格 JNI 从 ~5 次 + 多个临时对象 → ~1 次 + 零分配。消除本工程唯一运行时高频 GC 热点。
- **注意（后续维护）**：缓存的 `AndroidJavaObject`/`AndroidJavaClass` 不可 `using`/`Dispose`（需常驻）；静态字段在 app 生命周期内安全，Activity/Vibrator 同生命周期稳定，无需 OnDestroy 清理。

### 🟠 1.2 GraphicRaycaster 冗余 + 静态 TMP 开 RaycastTarget  ✅ 已实施
- **位置**：`GameManager.DisableUnneededRaycastTargets`（Start 中 ValidateReferences 之后调用）
- **改动**：采用代码方案而非改预制体——在 `Start` 统一关闭无交互 Graphic 的 `raycastTarget`：
  - `levelText` / `progressText` / `ruleTexts[0..2]` / `nextButtonText`（按钮子文字）的 `raycastTarget=false`。
  - 按钮自身的 Graphic **保留** `raycastTarget`（接收点击），仅关子文字。
  - 全屏 `Background` 的 `raycastTarget=false`：用 `FindTopmostCanvas` 从子 Canvas 向上回溯到根 Canvas（`gridParent.GetComponentInParent<Canvas>` 只能取到子 Canvas `Canvas_Static_Grid`，取不到根），再在根下按名找 `Background`。
- **关于 GraphicRaycaster 组件**：保留各 Canvas 的 raycaster 不删，仅让其下无交互 Graphic 的 RaycastTarget 关闭——等价零遍历开销，且不破坏兜底链路（删组件有"子 Canvas 偶发漏挂 raycaster 导致静默失点"风险，收益却相同，故不删）。
- **结果**：输入事件中无关 Graphic 不再参与 raycast 遍历与排序。改动可见可追溯，不依赖场景文件序列化值。

### 🟠 1.3 滑动震动密集调用，可能被系统合并/丢失
- **位置**：`HandlePointerEnter` → `FeedbackManager.Exclude()` → 每格震动
- **现象**：滑动每划入一格触发一次 Selection 震动。一秒划 20 格 = 20 次 `vibrate(createOneShot 20ms)`。
- **影响**：Android `Vibrator` 有最小触发间隔，密集调用可能被系统合并或丢弃；iOS `selectionChanged` 同理。马达过载可能产生"嗡"而非"嗒嗒嗒"。
- **建议**：对滑动震动加**时间节流**（如 40–60ms 内最多一次），或仅"状态真正改变"时震（当前 `changed` 已控音频，可同样控震动）。需真机实测手感后定阈值，**不强制**——属可选优化。

## 二、切关 / 载入性能

### 🟠 2.1 RoundedImage.OnPopulateMesh 每次 new List（切关 GC 峰）  ✅ 已实施
- **位置**：`RoundedImage.cs` 静态字段 `_arcPoints`
- **驱动**：规划支持 10×10=100 格。原 `new List<Vector2>(36)` 每次 populate 分配 ~336 字节，切关峰值 100×336≈34KB 一次性分配。改造后切关零分配。
- **改动**：新增 `static readonly List<Vector2> _arcPoints = new List<Vector2>(132)`（按 cornerSegments 最大 32 预分配，避免扩容）；`OnPopulateMesh` 内 `points = _arcPoints; points.Clear();` 复用，行为零变化（顶点坐标/顺序/UV/颜色全不变，仅存储容器从"每次 new"变"复用同一 List"）。
- **安全前提**：UGUI 重建在主线程同步顺序遍历 dirty Graphic、逐个调 `OnPopulateMesh`，**不重入**——一个未填完不会被另一个打断，故同一静态 List 可被各实例顺序复用。每次开头 `Clear()`（Count=0、capacity 保留），同帧内 Clear+填充+消费，不跨帧持有。
- **结果**：切关时 100 格 RoundedImage 重建的临时 List 分配降为零。

### 🟡 2.2 未烘焙关卡运行时同步解算
- **位置**：`SafeLoadLevelRoutine` → `MapSolver.Solve`（solutionData==null 时）
- **现象**：未烘焙的关在切关协程内**同步**回溯解算。
- **影响**：`MapSolver` 是回溯剪枝，小棋盘瞬时；但最坏复杂度高，大棋盘+无解时会卡主线程。当前关卡都建议已烘焙，回退罕见。
- **建议**：确保所有上线关卡都经 `Tools/Meowdoku/烘焙关卡正解` 烘焙（`solutionData` 非空）；运行时回退仅作兜底。无需改代码，运营流程保障即可。

### 🟡 2.3 启动同步 CSV 解析
- **位置**：`Start` → `LevelTableLoader.Load` + `RulesLoader.Load`
- **现象**：`Start` 同步读 CSV + 解析全部关卡。
- **影响**：关卡数少时无感；若关卡表很大，首帧卡顿。
- **建议**：当前规模无需改。若日后关卡表膨胀，可分帧解析或延迟到首关加载后异步预热后续关。低优先。

### 🟡 2.4 切关时 N 格 SetParent + GridLayoutGroup 布局
- **位置**：`SafeLoadLevelRoutine`（回收 `SetParent(null)` + 复用 `SetParent(gridParent)`）
- **现象**：切关时每格两次 `SetParent`，触发 GridLayoutGroup 重算。
- **影响**：N 格规模下单帧布局，16 格无感；大棋盘切关可能有小尖峰。
- **建议**：当前可接受。若需极致，可在切关期间临时禁用 `GridLayoutGroup.enabled`，手动按 cellSize 定位格子后再启用。低优先。

## 三、UI / 渲染

### 🟠 3.1 格子动效导致整个 Canvas_Static_Grid 每帧重提交
- **位置**：`TweenRunner.CatPop/CrossPunch/ShakeHorizontal/WaveJump` 改 transform
- **现象**：UGUI 同一 Canvas 合批为一个 mesh 提交；**任一子元素 transform 变化会让整个 Canvas 的 vertex buffer 重新上传 GPU**（不是 OnPopulateMesh 重建，是矩阵重算 + 整 buffer 重传）。16 格共享 `Canvas_Static_Grid`，无嵌套 Canvas。
- **影响**：
  - 叉号弹跳（单格 scale 动）→ `Canvas_Static_Grid` 每帧重传全部格子顶点（约 16×37 圆角 + GridBoard + 16×2×4 image ≈ 900 顶点）。当前规模轻。
  - **通关起舞时十几个 cat 同时动** → 每帧整 Canvas 重提交，持续 ~1s。这是动效最重时刻。
- **建议**：
  1. **优先观察**：用 Profiler 的 `Canvas.SendWillRenderCanvases` / `CanvasManager` 看起舞时是否成帧瓶颈。900 顶点重传一般 <0.2ms，多半可接受。
  2. 若成瓶颈：给"会动的节点"单独 Canvas。最省的做法是给 **cat 子节点**加一个小 Canvas（cat 是动效主体，起舞/破壳都动它），把动效的 Canvas 重提交隔离到 cat 自己的 Canvas，不影响其它格子。代价是 cat 与背景 RoundedImage 不再合批（多一个 draw call），但 16 格 draw call 增量小。
  3. 不建议给每格都加 Canvas（合批全毁，draw call 暴涨）。
- **结论**：当前规模大概率无需改；列入"若起舞卡顿再评估"。

### 🟡 3.2 RoundedImage cornerSegments 偏高  ✅ 已实施
- **位置**：`BlockPrefab` 背景 / `GridBoard` / `TopUI`（RuleBanner、RuleBlock0-2）
- **改动**：按圆角半径分级降段，平衡视觉与顶点量：
  - BlockPrefab 背景（radius=5，小格圆角）：8 → **4**（37→21 顶点/格，降 ~43%）
  - RuleBlock0/1/2（radius=10）：8 → **4**
  - RuleBanner（radius=15，大圆角折中）：8 → **6**
  - GridBoard（radius=12，大圆角折中）：8 → **6**
  - NextLevelButton（radius=0）：不动——RoundedImage 在 radius≤0.5 时回退普通矩形，segments 不参与，无需改
- **效果**：单格顶点 37→21（-43%）。10×10=100 格 + GridBoard + 4 TopUI 圆角，Canvas 重提交顶点总量显著下降，与 3.1 协同降低起舞/切关的重传开销。
- **风险**：纯视觉——圆角弧度变粗，4 段在 r=5 小格上通常可接受，6 段在 r=12/15 大圆角上折中。需 Unity 内肉眼确认圆角可接受；若某处明显多边形感，单独把那处段数上调即可（代码无依赖具体段数）。

### 🟡 3.3 TopUI 根孤儿 CanvasRenderer  ✅ 已实施
- **位置**：`TopUI.prefab` 根节点（fileID 1057189031069179240）
- **改动**：删除根节点上的 `CanvasRenderer` 组件（fileID 8608737344320297114）。已确认该 GameObject 仅挂 `RectTransform` + `CanvasRenderer` 两项，无任何 Graphic（Image/RoundedImage/TMP）——CanvasRenderer 是孤儿，无渲染者使用。
- **方式**：直接编辑预制体 YAML，删除 `m_Component` 列表里对该 CanvasRenderer 的引用 + CanvasRenderer 块本身。验证无悬挂引用（fileID 计数为 0），结构完整。
- **结果**：TopUI 根现仅留 RectTransform，无无效组件。无功能影响。

### 🟡 3.4 每格常驻 disabled 的 Outline 组件
- **位置**：`BlockController.Awake` → `EnsureErrorOutline` 给每格 `AddComponent<Outline>`（初始 disabled）
- **现象**：16 格各挂一个 disabled Outline。disabled 的 `ModifyMesh` 会 `IsActive()` 早退，渲染开销近零，但每格 populate 时仍参与 `GetComponents<IMeshModifier>` 遍历。
- **影响**：极小。但每格多一个组件占序列化/内存。
- **建议**：改为"双击点错时才 `AddComponent<Outline>`"（`LockAsError` 内按需添加），常态格子不挂 Outline。可减少组件数与遍历。低优先。

## 四、代码结构与可维护性（🔵 非性能）

### 🔵 4.1 BlockController 内混入动画触发与错误表现  ✅ 已实施
- **位置**：`BlockController.cs` / `GameManager.cs` 调用点
- **驱动**：即将新增动画，提前解耦以避免补丁机制（SilentBaseline）扩散。
- **改动**：显隐与动画职责分离——
  - `RefreshVisual` → `ApplyVisualState`：纯切 `Image.enabled`，无动画副作用。叉号隐藏时复位 `localScale`。
  - 新增显式动画方法：`PlayCatPop()`（破壳）、`PlayCrossPunch()`（出拳）、`PlayErrorShake()`、`LockAsError()`。
  - **删除 `SilentBaseline`**：显隐与动画解耦后，载入/given 只调 `ApplyVisualState`（不调动画方法）天然不播，不再需要"静默基线"补丁。
  - GameManager 6 处调用点改为"显隐 + 显式动画"组合，由调用方决定是否播动画。
- **行为等价**：逐场景核对（given 静默、滑动/单击/提交出拳、双击成功破壳、双击错误静默叉号+抖动）与重构前一致。叉号隐藏复位 scale、错误格不播出拳等边界均保持。
- **收益**：新增动画时直接在 BlockController 加 `PlayXxx()` 语义方法、在 GameManager 调用点显式编排，不再受过渡检测补丁纠缠。扩展面变干净。

### 🔵 4.2 静态单例 + 惰性创建的隐式依赖
- **位置**：`FeedbackManager.Instance`（惰性 getter）/ `TweenRunner.Host`（惰性）
- **现象**：单例在首次访问时 `new GameObject + AddComponent`。`FeedbackManager` 已用字段 `_instance` 规避递归（记忆已记录）。
- **建议**：当前实现正确。注意点：若未来加场景切换/重载，`DontDestroyOnLoad` 的单例与 `Instance` 惰性创建可能产生重复宿主——保持"Awake 内写字段、不读属性"的纪律即可。无需改。

### 🔵 4.3 硬编码数值散落
- **位置**：`TweenRunner` 各动画时长（0.06/0.10/0.20s 等）、`ShakeHorizontal` amplitude=14 freq=42、`WaveJump` height=30 reps=2、`RevealNextButtonAfter(0.5f)`
- **现象**：调优手感时需改代码重编译。
- **建议**：若策划需频繁调手感，可把这些时长/幅度集中到一个 `ScriptableObject`（`FeedbackTuning`）由 Inspector 配置，`TweenRunner` 读配置。属可扩展性优化，**当前规格已满足**，非必须。

### 🔵 4.4 RulesLoader 固定 3 条，却走 List 返回
- **位置**：`RulesLoader.Load` → `new List<string>(Rules)`
- **现象**：每次 `FillRuleTexts` new 一个 List（仅 Start 一次，低频）。
- **建议**：无需改（Start 一次性）。若洁癖可返回 `IReadOnlyList<string>` 复用静态数组。

## 五、逻辑 / 体验

### 🟠 5.1 通关后未锁定棋盘输入  ✅ 已实施
- **位置**：`GameManager` 新增 `isLevelCompleted` 字段
- **改动**：
  - `CheckRules` 胜利分支置 `isLevelCompleted = true`。
  - `HandlePointerDown` 首行守卫：通关后只给 `Tap()` 轻触反馈、不处理任何状态变更（含双击升级、滑动挂起）。
  - `SafeLoadLevelRoutine` 切关回收格子时复位 `isLevelCompleted = false`，新关卡解锁。
- **覆盖校验**：
  - 双击：`HandlePointerDown` 被锁 → 不会进入 `ExecuteDoubleClick`，无法制造新错误格 ✓
  - 滑动：`HandlePointerDown` 被锁时 `isPointerDown` 不置 true，`HandlePointerEnter` 首行 `!isPointerDown` 直接退出 → 滑动也被锁住 ✓
  - `HandlePointerClick`：本就无状态变更，无害 ✓
- **结果**：通关起舞/按钮弹出期间棋盘输入完全锁定（保留轻触反馈的一致性），不再与"已通关"状态冲突。

### 🟡 5.2 errorHintDuration 期间可继续操作
- **位置**：`ShowErrorHint` → `RestoreStatusAfter(errorHintDuration=1.5s)`
- **现象**：错误提示 1.5s 内 ProgressText 显示 "Not Cat!!!"，期间玩家若继续双击其它格，`UpdateStatusText` 会被新操作刷新覆盖提示（部分路径）或被 `errorHintRoutine` 恢复。
- **影响**：边缘体验，非 bug。
- **建议**：可选——错误提示期间锁 1.5s 输入或让提示不被覆盖。当前可接受，低优先。

### 🔴 7.1 Baker 解析逻辑重复 Loader  ✅ 已实施

- **结果**：CSV 格式变更只改一处，运行时读取与 Editor 烘焙自动一致，杜绝两份副本漂移。

## 附：无需动的"看着像问题其实不是"

- **滑动时 RoundedImage 重建**：滑动改的是 cat/cross（普通 Image）的 enabled，背景 RoundedImage 不重建。圆角不是滑动热点。
- **Image.enabled 反复设同值**：`changed=false` 时设 enabled 到相同值，UGUI 不触发 OnEnable/OnDisable，不重建。现有 `HandlePointerEnter` 无条件 `RefreshVisual` 安全。
- **TweenRunner._active 字典不清理**：对象池复用同一 Transform，key 被复用覆盖，不泄漏；协程结束不移除 key 是惰性，下次同 Transform 新补间时 `Kill` 覆盖。无实际泄漏。
- **FeedbackManager Editor Update 每帧**：`#if UNITY_EDITOR`，不进包，无运行时开销。
